Purpose of the Angular Roadmap

The Angular Roadmap is the official strategic communication layer between the Angular core team at Google and the global developer community. It serves three distinct functions:

🗺️

Transparency

Connects day-to-day GitHub commits to the framework's long-term vision — showing developers why decisions are made, not just what changed.

🎯

Strategic Alignment

Defines three pillars: AI developer experience, developer velocity, and raw framework performance — keeping the team focused and the community informed.

🤝

Community Contract

Features are listed by status (experimental → developer preview → stable), giving teams confidence about when to adopt new APIs in production.

The Three Strategic Goals (Angular v22+)

Goal 1
Improve the AI experience — AI-assisted coding, code generation quality, Dynamic UI generation (A2UI), MCP features, AI SDKs, and Gemini/Google AI Studio integrations.
Goal 2
Improve developer experience — Signal Forms to stable, tsgo compiler compatibility, Zoneless as default, OnPush as default CD strategy, Angular Aria stable, Vitest as primary test runner.
Goal 3
Improve framework performance — Zoneless removing zone.js overhead, incremental hydration (40-50% LCP gains), esbuild/Vite build pipeline (87% faster builds), tsgo compiler (5-10× TypeScript compile speedup).

Compiler Pipeline (tsgo era)

The most significant upcoming architectural change is the Angular compiler migrating to tsgo (TypeScript compiled to Go by Microsoft). This requires deep changes:

⚠ Impact
Angular has one of the deepest integrations with the TypeScript compiler of any framework. The tsgo migration will require the Angular compiler AND language service to be rebuilt — not just wrapper changes.
✓ Benefit
5–10× faster TypeScript compilation means Angular projects that take 30s to build today could complete in 3–6s. For large enterprise apps, this is transformational.

Framework Components

Angular's "components" in the architectural sense refers to the distinct system layers, not just UI components. Here's the complete map:

⚡

Signals (Reactivity Core)

signal(), computed(), effect(), linkedSignal() — all stable in v20. The new reactivity primitive replacing Zone.js + RxJS for most state.

🔁

Resource API

resource() and httpResource() for async signal-based data fetching. Currently experimental — expect stable in v22–v23.

📋

Signal Forms

Experimental in v21. Manages form state with Signals instead of FormControl observables. Interoperable with Reactive Forms for gradual migration.

🌊

Incremental Hydration

Stable in v20. Powered by @defer blocks. Hydrates parts of the page only when needed (on viewport, on interaction, etc.).

♿

Angular Aria (CDK)

8 headless, accessible interaction patterns launched in v21 developer preview. Bring your own styles, get WCAG-compliant interactions for free.

🧪

Vitest Integration

Stable unit test runner as of v21. Vitest replaces Karma as the primary runner. Migration schematic from Karma → Vitest available.

🛣️

Route-level Render Mode

Stable in v20. Declare per-route SSR/SSG/CSR strategy in the router config. Eliminates the need for separate apps or server configs per route type.

🔄

Event Replay (SSR)

Stable in v19, enabled by default in new projects. Captures user events during hydration and replays them — no more lost clicks on SSR pages.

Data Flow

Angular's data flow model has shifted from a push-based (Zone.js) model to a pull-based (Signals) model. Understanding both is essential for migrating existing apps.

Old Flow: Zone.js (Legacy)

// Zone.js patches async APIs — ANY async triggers CD on the whole tree
setTimeout(() => {
  // Zone.js intercepts this → marks the whole app as dirty → re-renders everything
  this.count++;
}, 1000);

New Flow: Signals + Zoneless (v20+)

// Signal graph: only components using this.count re-render
export class CounterComponent {
  count = signal(0);
  doubled = computed(() => this.count() * 2); // memoized

  increment() {
    this.count.update(n => n + 1);
    // Angular knows ONLY components depending on count need re-render
  }
}

// Async data: httpResource (experimental → stable soon)
products = httpResource<Product[]>('/api/products');
// products.value()  → signal with data
// products.status() → 'idle' | 'loading' | 'resolved' | 'error'
// products.error()  → error signal

Design Patterns

1. Smart/Dumb Component Pattern (Container/Presenter)

// SMART: owns state & logic
@Component({ standalone: true, template: `<user-list [users]="users()" (select)="onSelect($event)" />` })
export class UserPageComponent {
  users = httpResource<User[]>('/api/users');
  onSelect(user: User) { /* navigate */ }
}

// DUMB: only renders, no logic
@Component({ standalone: true, changeDetection: ChangeDetectionStrategy.OnPush })
export class UserListComponent {
  users = input.required<User[]>();
  select = output<User>();
}

2. Signal Store Pattern (State Management)

// NgRx Signal Store — replaces heavy NgRx boilerplate
export const UserStore = signalStore(
  withState({ users: [] as User[], loading: false }),
  withComputed(({ users }) => ({
    activeUsers: computed(() => users().filter(u => u.active))
  })),
  withMethods((store, http = inject(HttpClient)) => ({
    loadUsers: rxMethod<void>(pipe(switchMap(() => http.get('/api/users'))))
  }))
);

3. Deferred Loading Pattern

@defer (on viewport; prefetch on idle) {
  <!-- Heavy analytics dashboard, loads only when scrolled into view -->
  <analytics-dashboard />
} @loading (minimum 300ms) {
  <skeleton-loader />
} @error {
  <error-fallback />
}

4. Directive Composition Pattern (v15+)

@Component({
  hostDirectives: [
    { directive: CdkDrag },
    { directive: HasRipple, inputs: ['color'] },
    { directive: IsDisabled, outputs: ['disabledChange'] }
  ]
})
export class DraggableCardComponent {}

5. Route-level Render Strategy Pattern (v20+)

export const routes: Routes = [
  { path: 'home',      renderMode: RenderMode.Prerender },  // SSG
  { path: 'dashboard', renderMode: RenderMode.Client },     // CSR
  { path: 'product/:id',renderMode: RenderMode.Server },     // SSR

Security Considerations

Built-in Security Features (from Roadmap)

🛡️

Trusted Types API

Angular collaborates with Google Security team to support native Trusted Types, preventing DOM-based XSS at the platform level.

🔒

Auto-sanitization

Angular auto-sanitizes values bound to innerHTML, href, and src by default. Use DomSanitizer.bypassSecurityTrustHtml() only when explicitly safe.

🔐

Content Security Policy

Angular's Ivy renderer generates nonce-compatible style attributes, supporting strict CSP without unsafe-inline.

CSRF Protection

// Angular's HttpClient handles CSRF via the XSRF cookie pattern
provideHttpClient(
  withXsrfConfiguration({
    cookieName: 'XSRF-TOKEN',   // server sets this
    headerName: 'X-XSRF-TOKEN'  // Angular sends this
  })
)

Security Checklist

  • Never use bypassSecurityTrustHtml() on user-generated content
  • Use strict CSP with nonce — Angular's Ivy supports it natively
  • Enable HttpClient XSRF handling for state-mutating requests
  • Implement route guards with CanActivateFn for auth checks
  • Use Angular's built-in Trusted Types support (no manual DOM writes)
  • Audit third-party libraries — they bypass Angular's sanitization
  • Never store JWT in localStorage (XSS-accessible) — use httpOnly cookies
  • Never disable AOT — template compilation prevents injection attacks

Zoneless Security Benefit

ℹ Info
Removing zone.js also removes a subtle security surface: zone.js patches global async APIs (setTimeout, fetch, Promise) — any library that inspects those patches can fingerprint your app or inject malicious code into the async chain. Zoneless eliminates this attack vector entirely.

Production Best Practices

Build & Bundle

  • Use @angular-devkit/build-angular:application (esbuild builder)
  • Set build budgets in angular.json
  • Enable route-level code splitting via lazy routes
  • Use @defer for feature-level lazy loading
  • Enable compression (Brotli > gzip)

Change Detection

  • Use OnPush on ALL components (signals make this automatic)
  • Adopt Zoneless for new projects (stable v20.2)
  • Use computed() instead of getters in templates
  • Avoid calling functions in templates without memoization
  • Profile with Angular DevTools before optimizing

SSR / Hydration

  • Enable incremental hydration for content-heavy pages
  • Use isPlatformBrowser() for browser-only APIs
  • Enable event replay for better UX during hydration
  • Use route-level render mode — SSG for static, SSR for dynamic

Testing

  • Migrate from Karma to Vitest (schematic available)
  • Use Component Harnesses for component tests
  • Use TestBed with standalone components — no module setup
  • Mock signals with simple signal() values in tests

Monitoring in Production

// ErrorHandler for unhandled errors

export class GlobalErrorHandler implements ErrorHandler {

    // Send to Sentry, Datadog, etc.

Common Mistakes

✗ Mistake 1 — Calling Functions in Templates

Every change detection cycle calls functions in the template. This kills performance.

// BAD — getActiveUsers() called on every CD cycle
@for (user of getActiveUsers(); track user.id) {}

// GOOD — computed() is memoized, only recalculates when inputs change
activeUsers = computed(() => this.users().filter(u => u.active));
@for (user of activeUsers(); track user.id) {}
✗ Mistake 2 — Using Default Change Detection with Zone.js on Large Apps
Not enabling OnPush strategy means every async event re-renders the entire component tree. With 200+ components, this causes serious janks.
✗ Mistake 3 — Adopting Experimental APIs in Production
Signal Forms, httpResource, and Resource API are still experimental. Using them in production means you accept breaking API changes between minor versions.
✗ Mistake 4 — Memory Leaks from Manual Subscriptions
// BAD — subscription never cleaned up → memory leak
export class UserComponent implements OnInit {
  ngOnInit() { this.service.users$.subscribe(u => this.users = u); }
}

// GOOD — takeUntilDestroyed auto-unsubscribes on component destroy
users$ = this.service.users$.pipe(takeUntilDestroyed());
✗ Mistake 5 — Skipping Track in @for
Without track, Angular re-renders the entire list on every change instead of diffing by identity. Always use track item.id.
⚠ Mistake 6 — Not Migrating from NgModules
NgModules aren't removed, but all new Angular tooling, optimizations, and APIs are designed for standalone. Staying on NgModules means missing out on incremental hydration, signal inputs, and future compiler improvements.

Interview Questions

Beginner
What is the Angular Roadmap and what are its current three goals?
The roadmap is the Angular team's public communication of their strategic plans. The three goals are: (1) Improve AI developer experience, (2) Improve developer velocity/DX, and (3) Improve framework performance. These guide all feature work and aren't tied to specific version numbers.
Beginner
What is the difference between a "stable" feature and an "experimental" one in Angular?
Stable APIs follow semantic versioning — no breaking changes within a major version. Experimental APIs can have breaking changes in any release, including minors. Developer preview is between the two: mostly stable API but may have minor changes before going stable.
Intermediate
Explain Zoneless change detection and why it was added to the roadmap.
Zone.js works by monkey-patching all browser async APIs to trigger change detection. This is implicit, hard to debug, adds ~33KB to bundles, and prevents interop with non-Angular code. Zoneless Angular (stable v20.2) removes zone.js entirely, using Signals to track dependencies explicitly. Components only re-render when their signal dependencies change, giving finer-grained control, better performance, and smaller bundles.
Intermediate
How does incremental hydration differ from full hydration and what performance improvements does it bring?
Full hydration rehydrates the entire app on page load even if parts aren't interactive. Incremental hydration (powered by @defer blocks) only hydrates components when they're needed — on viewport entry, on interaction, on hover, etc. The Angular team measured 40-50% LCP improvements in real-world apps using incremental hydration vs CSR alone.
Intermediate
What problem does the tsgo compiler integration solve?
Microsoft ported the TypeScript compiler to Go (tsgo), promising 5–10× compilation speedup. Angular has deep TypeScript compiler integration (template type-checking, language service, etc.), so supporting tsgo requires major architectural changes to the Angular compiler. The benefit is dramatically faster build and type-check times for large Angular projects.
Expert
How would you architect a migration strategy from Zone.js to Zoneless for a large enterprise Angular app?
Phase 1: Enable OnPush on all components to identify components relying on Default CD. Phase 2: Replace all @Input/@Output with signal-based input()/output(). Phase 3: Migrate services from BehaviorSubject to signal(). Phase 4: Add provideExperimentalZonelessChangeDetection() alongside zone.js in parallel (Angular supports running both). Phase 5: Fix components that fail in Zoneless (usually those manually calling cdr.markForCheck() incorrectly). Phase 6: Remove zone.js from polyfills.ts. The key risk is third-party libraries that depend on zone.js — audit them first.
Expert
Explain the signal graph model and how computed() achieves memoization without a framework scheduler.
Angular's signal implementation uses a reactive graph where each computed() node tracks which signals it read during its last execution (its "dependencies"). When a source signal changes, it marks dependent computed signals as "dirty" but doesn't recompute them. The computed value is only recalculated lazily when it's read next time — this is "glitch-free lazy evaluation." The Angular runtime's change detection integrates with this graph: a component is only scheduled for re-render if a signal it reads (directly in its template or via computed()) has been dirtied. No zone, no scheduler, no explicit subscriptions.

Learning Notes by Level

🟢 Beginner Notes
  • Standalone components are the default: You no longer need NgModules. Every new Angular project generates standalone components — don't look for AppModule, it's gone.
  • Signals replace most RxJS usage: For local state and simple async, use signal(), computed(), and effect(). Only reach for RxJS when dealing with complex stream operations (debounce, merge, race).
  • New control flow syntax (@if, @for, @switch): The *ngIf, *ngFor structural directives still work, but the new built-in control flow is cleaner, more performant, and is the Angular team's recommendation for all new code.
  • The roadmap uses status labels — follow them: If something is "experimental," don't use it in your job's production app. Use "stable" APIs for production and "developer preview" with care.
  • angular.dev is the official docs site: angular.io is legacy. All new Angular v17+ documentation is at angular.dev.
🟡 Intermediate Notes
  • OnPush is the performance baseline: Enable OnPush on every component as a default. With signals, Angular automatically marks only signal-dependent components for re-render. Without OnPush, you're leaving performance on the table.
  • Route-level render modes replace universal/app-shell complexity: Instead of configuring separate SSR/CSR apps, declare renderMode per route in your router config. This is cleaner, more maintainable, and supported by the Angular CLI build pipeline.
  • @defer is your bundle-splitting tool for templates: Anything below the fold, modals, drawers, analytics — wrap in @defer. It generates separate code chunks automatically without manual dynamic import() calls.
  • Event replay solves the SSR "lost click" problem: Users on slow connections often click before hydration completes. withEventReplay() queues those events and replays them post-hydration. Enable it in all SSR apps.
  • The Angular Aria CDK patterns are for building design systems: If your team uses a custom component library, Angular Aria gives you accessible keyboard interactions, ARIA roles, and focus management out of the box — you style it yourself.
🔴 Expert Notes
  • tsgo integration will require compiler re-architecture: The Angular Language Service and template type-checker are deeply integrated with the TypeScript compiler's internal APIs. tsgo uses a different Go-native API surface. Watch the angular/angular repo for the RFC — when it lands, your custom schematics, eslint-plugin-angular, and ng-packagr configs may need updates.
  • Signal Forms' interop model is the key migration story: The planned interop between Signal Forms and Reactive Forms means you can migrate form by form. Understand FormGroup → Signal Form bridge APIs before the stable release so you can plan migration at scale.
  • httpResource is replacing the Service + Observable pattern, but not fully yet: httpResource lacks operators (retry, debounce, switchMap) that RxJS handles elegantly. For complex data flows, combine: use httpResource for simple fetches, RxJS resource() factory for complex reactive chains, and NgRx Signal Store for shared server state.
  • AI code generation quality for Angular is already high (per roadmap): The Angular team's Web Codegen Scorer benchmarks show modern LLMs generate correct Angular code. The remaining gaps are in library-specific patterns (NgRx, Angular Material customization tokens) and cutting-edge APIs (Signal Forms, httpResource). You can improve LLM output by including framework version in your prompts and linking to specific angular.dev guide URLs.
  • Zoneless + Incremental Hydration is the performance ceiling for Angular apps: An app using Zoneless (no zone.js overhead), OnPush everywhere (fine-grained CD), Signals (lazy memoized graph), @defer (split bundles), incremental hydration (progressive activation), and prerendered static routes represents the current theoretical best case for Angular performance on the web.
